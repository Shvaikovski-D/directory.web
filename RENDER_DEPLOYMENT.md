## Деплой на Render.com

### Шаг 1: Пуш изменений в GitHub

Убедитесь, что все изменения закоммичены и запушены в ваш GitHub репозиторий:

```bash
git add .
git commit -m "Настройка для деплоя на Render.com"
git push origin main
```

### Шаг 2: Подключение репозитория в Render

1. Перейдите на [dashboard.render.com](https://dashboard.render.com/)
2. Войдите или создайте аккаунт (если используете GitHub - войдите через GitHub)
3. Нажмите кнопку **"New +"** в правом верхнем углу
4. Выберите **"Web Service"**

### Шаг 3: Настройка Web Service

#### 3.1. Подключение репозитория

1. Нажмите **"Connect GitHub"** (если ещё не подключён)
2. Разрешите доступ к вашим репозиториям
3. Выберите репозиторий `directory.web`
4. Выберите ветку для деплоя (обычно `main` или `master`)

#### 3.2. Настройка Docker

В разделе **"Build & Deploy"**:

- **Runtime**: `Docker`
- **Docker Context**: `.` (корневая директория)
- **Dockerfile Path**: `./Dockerfile`
- **Publish Directory**: `/app` (это значение по умолчанию, изменять не нужно)

#### 3.3. Настройка Health Check

⚠️ **Важно**: В разделе **"Advanced"** нужно настроить Health Check:

1. Найдите блок **"Health Check"**
2. **Health Check Path**: `/health`
3. **Health Check Timeout**: можно оставить `30s` (по умолчанию)
4. **Health Check Interval**: можно оставить `10s` (по умолчанию)
5. **Health Check Grace Period**: можно оставить `0s` (по умолчанию)

Render.com будет автоматически проверять endpoint `/health` каждые 10 секунд. Если приложение отвечает статусом 200 OK, считается что сервис работает корректно.

#### 3.4. Переменные окружения

⚠️ **Важно**: В разделе **"Environment Variables"**:

Обязательные переменные:

| Имя переменной | Значение | Описание |
|----------------|----------|----------|
| `ASPNETCORE_ENVIRONMENT` | `Production` | Режим работы приложения (опционально, но рекомендуется) |

Не устанавливайте переменную `PORT` - Render.com устанавливает её автоматически!

**Примечание по базе данных**:
Connection string к PostgreSQL уже настроен в файле `src/Web/appsettings.json`. Дополнительные переменные окружения для базы данных не требуются.

Если вы хотите переопределить connection string через Render, добавьте переменную:
- `ConnectionStrings__directory.webDb` = `ваша_connection_string`

#### 3.5. План и регион

- **Plan**: Выберите подходящий план:
  - **Free** (бесплатный, но с ограничениями)
  - **Starter** (~$7/месяц)
  - **Standard** (~$25/месяц)
  - и т.д.

- **Region**: Выберите регион, ближайший к вашей базе данных PostgreSQL
  - Для вашей базы данных (Frankfurt) выберите **Frankfurt (EU Central)**

#### 3.6. Имя сервиса

- **Name**: Введите имя для вашего сервиса (например, `directory-web-api`)
- Это имя станет частью URL: `https://directory-web-api.onrender.com`

### Шаг 4: Создание и деплой

1. Нажмите кнопку **"Create Web Service"** в нижней части страницы
2. Render автоматически начнёт сборку Docker образа
3. Процесс деплоя может занять 2-5 минут

### Шаг 5: Проверка деплоя

После успешного деплоя:

1. Вы увидите зелёный статус **"Live"**
2. Нажмите на URL вашего сервиса (например, `https://directory-web-api.onrender.com`)
3. Вы будете перенаправлены на страницу Scalar API документации

**Проверьте Health Check**:
```
https://directory-web-api.onrender.com/health
```

Должен вернуться ответ:
```json
{"status":"Healthy"}
```

## Возможные проблемы и решения

### Проблема: Health Check failing

**Симптом**: Render показывает статус unhealthy

**Решение**:
1. Проверьте логи в Render dashboard
2. Убедитесь, что endpoint `/health` доступен:
   ```bash
   curl https://ваш-сервис.onrender.com/health
   ```
3. Проверьте, что приложение запускается на правильном порту

### Проблема: Ошибка подключения к базе данных

**Симптом**: Приложение запускается, но не может подключиться к PostgreSQL

**Решение**:
1. Проверьте connection string в `appsettings.json`
2. Убедитесь, что IP-адреса Render разрешены в настройках PostgreSQL
3. Проверьте логи для более детальной информации об ошибке

### Проблема: Очень медленный деплой

**Симптом**: Дескрипт занимает более 10 минут

**Решение**:
1. Проверьте размер Docker образа (можно оптимизировать Dockerfile)
2. Убедитесь, что используется кэширование слоёв Docker
3. Рассмотрите использование paid плана для более быстрых CPU

### Проблема: App crashed на старте

**Симптом**: Приложение запускается, но сразу падает

**Решение**:
1. Проверьте логи startup errors
2. Убедитесь, что все зависимости установлены
3. Проверьте, что порт не используется другим сервисом
